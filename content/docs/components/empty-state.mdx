## Usage

Recommendations

  

##### Do

*   **Do** make it clear if this is an expected situation or an error.
    
*   **Do** provide a next step for the user to take to resolve the situation if possible.
    

##### Don't

*   **Don't** steal visual focus from the rest of the page.

Anatomy

1.  (Optional) Icon to help communicate the message.
    
2.  (Optional) Give a quick overview of the empty state message.
    
3.  Give details of why the empty state is present.
    
4.  (Optional) Actions that can be taken on the empty state. 
    

  


Alignment

To prevent the empty state from jumping when switching to the progress or error state, define a container that is centered and align the empty state within it, ensuring that the layout remains consistent during state transitions.

<imagePlaceholder imageId="cv-empty-state_transitions" />


Button usage

Follow the rule of keeping only one primary button on each page when possible. If there is no primary button on the page, a primary button may be used in the empty state. If one exists already, use an outlined button or text button for less emphasis. 


Content guidelines

Explain to the user that no data is available. Make it clear if this is an expected situation or an error. Provide a next step for the user to take to resolve the situation.

## Accessibility

Placeholder content displayed when there is no data to show.


Overview

Empty states are designed to be accessible to everyone. They are built to ensure the UI state is understandable and actionable.


Keyboard interactions

To support accessibility through keyboard navigation, empty states include the following interactions:


Tab index focus

Ensures that the action buttons in empty states are included in sequential keyboard navigation, allowing users to reach and interact with them using the tab key.


Hit area state

A visible focus indicator highlights the action buttons when they gain keyboard focus, ensuring users can easily identify which element is active.
